# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

Personal editor configuration shared across **Vim**, **Neovim**, **IdeaVim** (JetBrains IDEs), and **VSCode-Neovim**. The repo lives at `~/work/vimrc` and `scripts/install.sh` symlinks its files into `~/.vimrc`, `~/.ideavimrc`, and `~/.config/nvim` so editing here = editing live config.

## Use case and priorities

Neovim is used here primarily for quick ad-hoc file edits in the terminal and as the preferred `$EDITOR` / `$VISUAL`, **not** as a full IDE replacement — heavy development and debugging happen elsewhere (typically the JetBrains IDE driving `.ideavimrc`). Optimize this config for **simplicity, startup speed, and long-term stability** over feature breadth. The config is touched infrequently, so silent breakage between sessions is more costly than missing features: prefer plugins with stable APIs and core/built-in alternatives over churn-prone ones, and don't add machinery that isn't earning its keep.

## Layered configuration architecture

Each editor sources a different entry point, but they share the common base in `.vimrc`. Understand the layering before editing:

- **`.vimrc`** — base settings + mappings shared by *all* editors. Anything universal (leader, search, indent, leader-mappings like `<leader>h`, `<leader>d`) belongs here. It uses `if !has('ide')` to skip pieces that break in IdeaVim.
- **`.ideavimrc`** — `source ~/.vimrc` + IdeaVim-only `Plug` lines and `<Action>` mappings that delegate to JetBrains IDE actions (e.g. `<leader>pb` → `CidrBuildRunToolbar`, `gs` → `CIDR.Lang.SwitchHeaderSource`). Don't add Lua here — IdeaVim doesn't run it.
- **`init.lua`** (Neovim) — sources `~/.vimrc`, bootstraps `lazy.nvim`, then `require("lazy").setup("plugins")`.
- **`vs-code-init.lua`** + **`vs-code-vimrc`** — separate, intentionally minimal stack for the VSCode-Neovim extension. Don't try to merge it with the main Neovim path; VSCode hosts its own UI so most plugins/mappings are inappropriate.

When adding behavior, pick the *narrowest* layer that works — putting an editor-specific keymap in `.vimrc` will leak into IdeaVim/VSCode and likely break.

## Neovim plugin system (Lazy.nvim)

- Plugin specs live in **`lua/plugins/`**. `require("lazy").setup("plugins")` loads `lua/plugins/init.lua` *and* every other `*.lua` file in that directory automatically — each file returns a spec table. Each plugin gets its own file (`treesitter.lua`, `neo-tree.lua`, `oil.lua`, `gitsigns.lua`, `lsp.lua`, `cmp.lua`, `marks.lua`, `lualine.lua`, `which-key.lua`, `telescope.lua`, `harpoon.lua`, `lazydev.lua`); `lua/plugins/init.lua` only holds bare entries that need no `config` callback (plenary, nvim-surround). Follow this pattern: add a new file, do not append to `init.lua`.
- LSP servers are enabled via `vim.lsp.enable({ 'clangd', 'cmake', 'rust_analyzer', 'lua_ls' })` inside the `config` callback in `lua/plugins/lsp.lua` — that's the source of truth for which language servers run. The shared `LspAttach` autocmd that wires up buffer-local keymaps (`gd`, `K`, `gr`, `<space>rn`, etc.) lives there too.
- `lazy-lock.json` pins commits — commit it when you change plugins.
- `.luarc.json` (under `lua/`) is gitignored; it's auto-generated by `lazydev.nvim` (the successor to `neodev.nvim`).

## Common operations

- **Update plugins**: open Neovim, run `:Lazy sync`. Commit the resulting `lazy-lock.json` change.
- **Install missing tree-sitter parsers**: handled automatically by `nvim-treesitter` (`auto_install = true`).
- **Install LSP servers**: managed via Homebrew (see `README.md`); Mason was intentionally removed.
- **Apply local edits**: because installation symlinks the repo into `~/.config/nvim` etc., changes take effect on next editor start (or `:source %`) — no copy step.
- **Fresh install on a new machine**: `bash scripts/install.sh` (clones to `~/work/vimrc` and creates symlinks).

There is no test suite, build step, or linter wired up.

## Conventions worth knowing

- Leader is `<space>`. Plugin keymaps cluster by prefix: `<leader>f*` Telescope, `<leader>g*` gitsigns hunks, `<leader>c*` LSP code (rename/action/format), `<leader>w*` LSP workspace, `<leader>p*` IDE project actions (IdeaVim only), `<leader>n/r/b` neo-tree views. `<leader>h` stays a leaf for `:noh` (defined in `.vimrc`).
- **Two file managers on purpose, do not consolidate**: `neo-tree.nvim` is a persistent sidebar tree for browsing project structure (`<leader>n`/`<leader>r`/`<leader>b`); `oil.nvim` is a directory-as-buffer editor for fast in-place rename / create / delete (`-` opens the parent directory). Different workflows, both kept deliberately.
- Every `vim.keymap.set(...)` call should include `{ desc = "..." }` — `:WhichKey` and the leader popup use it as the label, otherwise they fall back to showing the raw RHS function. Prefix groups themselves are registered in `lua/plugins/which-key.lua`'s `spec`.
- `gs` is overloaded on purpose: in Neovim it calls `ClangdSwitchSourceHeader`; in IdeaVim it triggers the equivalent JetBrains action. Keep them in sync if rebinding.
- The Russian `langmap` line in `.vimrc` lets normal-mode commands work with a Russian keyboard layout — don't strip it.
