# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

Personal editor configuration shared across **Vim**, **Neovim**, **IdeaVim** (JetBrains IDEs), and **VSCode-Neovim**. The repo lives at `~/work/vimrc` and `scripts/install.sh` symlinks its files into `~/.vimrc`, `~/.ideavimrc`, and `~/.config/nvim` so editing here = editing live config.

## Layered configuration architecture

Each editor sources a different entry point, but they share the common base in `.vimrc`. Understand the layering before editing:

- **`.vimrc`** — base settings + mappings shared by *all* editors. Anything universal (leader, search, indent, leader-mappings like `<leader>h`, `<leader>d`) belongs here. It uses `if !has('ide')` to skip pieces that break in IdeaVim.
- **`.ideavimrc`** — `source ~/.vimrc` + IdeaVim-only `Plug` lines and `<Action>` mappings that delegate to JetBrains IDE actions (e.g. `<leader>pb` → `CidrBuildRunToolbar`, `gs` → `CIDR.Lang.SwitchHeaderSource`). Don't add Lua here — IdeaVim doesn't run it.
- **`init.lua`** (Neovim) — sources `~/.vimrc`, bootstraps `lazy.nvim`, then `require("lazy").setup("plugins")` and `require("config")`.
- **`vs-code-init.lua`** + **`vs-code-vimrc`** — separate, intentionally minimal stack for the VSCode-Neovim extension. Don't try to merge it with the main Neovim path; VSCode hosts its own UI so most plugins/mappings are inappropriate.

When adding behavior, pick the *narrowest* layer that works — putting an editor-specific keymap in `.vimrc` will leak into IdeaVim/VSCode and likely break.

## Neovim plugin system (Lazy.nvim)

- Plugin specs live in **`lua/plugins/`**. `require("lazy").setup("plugins")` loads `lua/plugins/init.lua` *and* every other `*.lua` file in that directory automatically — each file returns a spec table. Add a new file rather than appending everything to `init.lua`.
- **`lua/config.lua`** is legacy: a single setup file from the pre-Lazy/Packer era containing `mason`, `treesitter`, `neo-tree`, `gitsigns`, `cmp`, `marks`, and the LSP `LspAttach` autocmd with all buffer keymaps. The file's own header comment notes it should be refactored into per-plugin spec files; new plugin setup should go in `lua/plugins/<name>.lua` (see `telescope.lua`, `dap.lua`, `harpoon.lua` for the pattern: spec table with `init`/`config` callbacks).
- LSP servers are enabled via `vim.lsp.enable({ 'clangd', 'cmake', 'rust_analyzer', 'lua_ls' })` in `config.lua` — that's the source of truth for which language servers run.
- `lazy-lock.json` pins commits — commit it when you change plugins.
- `.luarc.json` (under `lua/`) is gitignored; it's auto-generated by `neodev.nvim`.

## Common operations

- **Update plugins**: open Neovim, run `:Lazy sync`. Commit the resulting `lazy-lock.json` change.
- **Install missing parsers/LSPs**: handled automatically by `nvim-treesitter` (`auto_install = true`) and `mason`.
- **Apply local edits**: because installation symlinks the repo into `~/.config/nvim` etc., changes take effect on next editor start (or `:source %`) — no copy step.
- **Fresh install on a new machine**: `bash scripts/install.sh` (clones to `~/work/vimrc` and creates symlinks).

There is no test suite, build step, or linter wired up.

## Conventions worth knowing

- Leader is `<space>`. Plugin keymaps cluster by prefix: `<leader>f*` Telescope, `<leader>h*` gitsigns hunks, `<leader>p*` IDE project actions (IdeaVim only), `<leader>n/r/b/g` neo-tree views.
- `gs` is overloaded on purpose: in Neovim it calls `ClangdSwitchSourceHeader`; in IdeaVim it triggers the equivalent JetBrains action. Keep them in sync if rebinding.
- The Russian `langmap` line in `.vimrc` lets normal-mode commands work with a Russian keyboard layout — don't strip it.
